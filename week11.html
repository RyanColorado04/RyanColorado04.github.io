Chapter 3 Notes:

Functional programs donâ€™t update variables or modify mutable data structures. This chapter shows what type of data structures you can use.

A functional data structure is operated on using only pure functions. Pure functions do not change data in place or perform other side effects. Thus, functional data structures are by definition immutable.

The structure that this chapter uses as an example is the linked list. I will post the full data structure later. The data structure is defined like this:

Sealed trait List [+A]

In general, a data type is introduced with the trait key word. A trait is an abstract interface that may optionally contain implementations of some methods. In this case, a trait List is declared with no methods. The sealed keyword in front means that all implementations of the trait must must be declared in this file. Note, abstract class could also have been used here instead of trait.

The + in front of the type parameter A is a variance annotation that signals that A is a covariant or positive parameter. This lets it work with subtypes in Scala.

This file also contains two implementations, or data structures, of List.
Example: case object Nil extends List[Nothing]

Using the type parameter [+A] after sealed trait List makes this data type polymorphic. We can use this same definition for a list of integers, doubles, strings, etc.
Example: val ex1: List Double = Nil
Example 2: List[Int] = Cons(1, Nil)

The data structure also uses pattern matching:


def sums(ints: List[Int]): Int = ints match {
	case Nil => 0
	case Cons(x, xs) => x + sum(xs)
}

Pattern matching in Scala is like a fancy switch statement that may descend into the structure of the expression it examines and extract subexpressions of that structure. If the target matches the pattern in a case, the result of that case becomes the result of the entire match expression.

The chapter then goes over adding or removing elements from a list if the data is immutable. This is surprisingly simple. For example, when you add an element 1 to the front of an existing list, a new list is returned. Since lists are immutable, it is just reused in a process called data sharing. This lets functions be implemented more efficiently. You never have to worry about subsequent code later modifying your data later. This is a benefit of functional programming. Writing purely functional data structures that support different operations efficiently is all about finding clever ways to exploit data sharing.

This chapter also goes over foldRight briefly. I will add notes on this in a later section.

This chapter also briefly goes over using underscore notation for anonymous functions. Example:


(x, y) => x + y

This can be rewritten as:


(x, y) => _ + _

Underscore notation can be used in situations where they type of x and y can be inferred by Scala. This is useful in examples like foldRight and pattern matching.

The chapter also mentions that List exists in the standard Scala library. The rest of the book, the authors use the Scala standard version.
